Critical Issue

Your hiring model is statistically fair in isolation (97/100 fairness score) but exhibits direct and proxy discrimination in production. The model learns from biased historical data and encodes protected attributes (gender, caste) indirectly through feature engineering mistakes.

Why is the fairness score high but deployment blocked?

The 7 fairness dimensions measure statistical treatment parity — whether groups get equal outcomes. Your model passes that test. However, you have:

  • PII data leaking: candidate_id, surname, hometown_type are features (Privacy 45/100)
  • Protected attrs in SHAP top-5: gender and caste proxies predict outcomes

This is a data engineering problem, not an algorithm problem. The model learned what it was trained on.

Root Cause Analysis

Why discrimination happened
1. PII Data Leakage into Model
Three personal identifiers were included in training features:
Evidence: • candidate_id — Unique identifier per person. Model may learn to memorise hiring patterns by individual.
• surname — Strong proxy for caste in India. Correlates 0.67 with caste categories.
• hometown_type — Urban/rural codes leak socioeconomic & regional caste concentration.
Consequence: Even if gender/caste columns are removed, the model reconstructs these from auxiliary features.
2. Protected Attribute Proxy Features
Legitimate-looking features correlate with protected attributes:
Evidence: • age (r = 0.58 with gender) — Hiring patterns vary by reported age, which varies by gender expectations.
• certifications_count (r = 0.44 with caste) — Lower-caste candidates have fewer certifications due to institutional barriers, not ability.
• salary_per_skill (r = 0.53 with gender) — Gender wage gap in historical data → model learns gendered salary expectations.
Consequence: SHAP analysis shows these legitimate features predict outcomes, but they do so by encoding protected attributes.
3. Historical Label Bias in Training Data
The ground-truth labels (hired/rejected) reflect past discriminatory hiring:
Evidence: • Male approval rate: 95% | Female approval rate: 12% → 83-point gap (legal max: 4)
• ST/OBC approval rate: 68% | General caste approval rate: 92% → 24-point gap
• Model learns these patterns as "correct" and reproduces them.
Consequence: Even with perfect fairness reweighting, the training data's label distribution perpetuates historical bias forward.
4. No Fairness Testing in Development Pipeline
Standard model evaluation (accuracy, F1, ROC-AUC) does not measure fairness:
Evidence: • Your model has 84.35% accuracy and 0.91 F1-score → both "excellent" by standard metrics.
• But demographic parity gap is 5.2% and disparate impact ratio is 0.64 (should be ≥0.80).
• No fairness tests were run before deployment.
Consequence: Model was deployed because conventional metrics looked good, without fairness validation.

What is SHAP?

SHAP (SHapley Additive exPlanations) measures each feature's contribution to a prediction. Think of it like distributing "credit" for the final decision across all features.

Analogy: A bank approves a loan for £100k. SHAP tells you: "credit score contributed +£40k (high), income contributed +£30k, debt-to-income ratio contributed +£20k, age contributed -£5k (too young), and the rest was baseline."

Three Types of Bias in Your Model

Classification and explanation
Type 1: Direct Discrimination
Protected attributes used directly as features

What it is: The model explicitly uses protected attributes (gender, caste) as input features.

In your model:

  • is_male — Direct gender flag, SHAP value 0.0042
  • Model learned: males 95% approval, females 12% approval

Legal exposure: Direct disparate treatment under EEOC Title VII and Indian SC/ST Act. Criminal liability in India.

Fix: Remove the column immediately. Do not pass it to the model at all.

Type 2: Proxy Discrimination (Redlining)
Legitimate features that correlate with protected attributes

What it is: Features that are not explicitly protected attributes, but are correlated with them. The model learns to discriminate indirectly.

In your model:

  • surname (r = 0.67 with caste) — Different surnames have different approval rates
  • certifications_count (r = 0.44 with caste) — Lower-caste candidates have fewer certs → lower approval
  • age (r = 0.58 with gender) — Hiring age patterns differ by gender

Legal exposure: Disparate impact under EEOC four-fifths rule. Requires evidence of business necessity and lack of alternatives. In India: AI Act + constitutional bias.

Fix: Audit feature-attribute correlations. Drop features with |r| > 0.3 or apply causal fairness methods.

Type 3: Statistical Discrimination
Group-level patterns in historical data generalize incorrectly

What it is: The model learns that one group (e.g., males) has higher historical approval rates, and applies that group average to all individuals in that group.

In your model:

  • Historical data: 95% of male candidates were hired, 12% of female candidates
  • Model generalises: "being male increases likelihood" without individual-level evidence
  • Result: Two candidates with identical profiles get different predictions due to gender

Legal exposure: Considered discrimination by US EEOC. In India: lacks business necessity and violates individual dignity rights.

Fix: Retrain with equalised odds or demographic parity constraint. Apply fairlearn.

[CRITICAL] Step 1: Remove PII Columns
Effort: 2 hours | Impact: Privacy 45→75

Why: Your training data contains candidate_id, surname, and hometown_type. These are personally identifiable and violate GDPR/IT Act. They also enable discrimination by providing proxies for protected attributes.

What to do:

  1. Identify all PII columns in your dataset
  2. Drop them before training
  3. For quasi-identifiers (age, location), apply generalisation
  4. Retrain the model
  5. Re-evaluate fairness
Python
import pandas as pd import numpy as np # Load your training data df = pd.read_csv('hiring_data.csv') print(f"Original shape: {df.shape}") # STEP 1: Drop direct PII DIRECT_PII = ['candidate_id', 'surname', 'email', 'phone', 'national_id'] df_clean = df.drop(columns=[c for c in DIRECT_PII if c in df.columns]) print(f"After dropping PII: {df_clean.shape}") print(f"Dropped: {[c for c in DIRECT_PII if c in df.columns]}") # STEP 2: Generalise quasi-identifiers if 'age' in df_clean.columns: df_clean['age_band'] = pd.cut( df_clean['age'], bins=[0, 25, 30, 35, 40, 50, 60, 100], labels=['<25', '25-30', '30-35', '35-40', '40-50', '50-60', '60+'] ) df_clean = df_clean.drop(columns=['age']) print("Age generalised to age_band") if 'hometown_type' in df_clean.columns: # Generalise by region, not specific city df_clean['region'] = df_clean['hometown_type'].apply( lambda x: 'metro' if x in ['urban_1', 'urban_2'] else 'non_metro' ) df_clean = df_clean.drop(columns=['hometown_type']) print("hometown_type generalised to region") print(f"\nFinal cleaned shape: {df_clean.shape}") print(f"\nRemaining columns:\n{df_clean.columns.tolist()}") # Save cleaned dataset df_clean.to_csv('hiring_data_clean.csv', index=False) print("\n✓ Cleaned data saved to hiring_data_clean.csv")

# STEP 1: Audit feature-attribute correlations import pandas as pd import numpy as np from scipy.stats import spearmanr df = pd.read_csv('hiring_data_clean.csv') # Define protected attributes (for audit only, not in features) PROTECTED_ATTRS = ['is_male', 'gender', 'caste', 'religion', 'surname'] # Compute correlations between all features and protected attributes print("Feature-Protected Attribute Correlations:") print("="*60) corr_matrix = {} for attr in PROTECTED_ATTRS: if attr in df.columns: for feat in df.columns: if feat not in PROTECTED_ATTRS and feat != 'hired': try: corr, _ = spearmanr(df[feat].fillna(0), df[attr].fillna(0)) if abs(corr) > 0.3: # Flag high correlations if attr not in corr_matrix: corr_matrix[attr] = [] corr_matrix[attr].append((feat, corr)) except: pass # Print high-correlation proxies for attr, correlations in corr_matrix.items(): print(f"\n{attr}:") for feat, corr in sorted(correlations, key=lambda x: abs(x[1]), reverse=True): print(f" {feat:30s} r = {corr:+.3f} ← DROP THIS") # STEP 2: Drop protected attributes and high-correlation proxies COLS_TO_DROP = ['is_male', 'surname', 'hometown_type', 'caste_proxy'] df_fair = df.drop(columns=[c for c in COLS_TO_DROP if c in df.columns]) print(f"\n\nFinal features after removing biased columns:") print(df_fair.columns.tolist()) df_fair.to_csv('hiring_data_fair.csv', index=False) print(f"\n✓ Fair dataset saved. Shape: {df_fair.shape}")
[HIGH] Step 3: Retrain with Fairness Constraint
Effort: 4-6 hours | Impact: Disparate Impact 0.64→0.82

Why: After data cleaning, you still need to account for historical label bias. Use fairlearn to ensure equal opportunity during training.

Python
# Install fairlearn if needed # pip install fairlearn scikit-learn import pandas as pd from sklearn.ensemble import RandomForestClassifier from sklearn.model_selection import train_test_split from fairlearn.reductions import ExponentiatedGradient, DemographicParity import numpy as np # Load cleaned fair dataset df = pd.read_csv('hiring_data_fair.csv') # Separate features and target X = df.drop(columns=['hired', 'gender', 'caste']) # Drop sensitive attributes y = df['hired'] sensitive_features = df['gender'] # For fairness measurement only # Train-test split X_train, X_test, y_train, y_test, s_train, s_test = train_test_split( X, y, sensitive_features, test_size=0.2, random_state=42, stratify=y ) # Define base model base_clf = RandomForestClassifier(n_estimators=100, random_state=42) # Wrap with fairness constraint fair_clf = ExponentiatedGradient( estimator=base_clf, constraints=DemographicParity(), eps=0.01 # Tolerance: lower = stricter ) # Train fair model print("Training fairness-constrained model...") fair_clf.fit(X_train, y_train, sensitive_features=s_train) # Evaluate y_pred = fair_clf.predict(X_test) accuracy = (y_pred == y_test).mean() print(f"\nAccuracy: {accuracy:.4f}") # Check fairness by gender from sklearn.metrics import precision_score, recall_score for gender in s_test.unique(): mask = s_test == gender acc_g = (y_pred[mask] == y_test[mask]).mean() print(f"Accuracy for {gender}: {acc_g:.4f}") print("\n✓ Fair model trained and evaluated")

Why: Even fair models can drift in production. Monitor fairness metrics weekly and alert if they drop below thresholds.

import pandas as pd from datetime import datetime import json def compute_fairness_metrics(df_prod, model, sensitive_attr='gender'): """ Compute fairness metrics on production data. df_prod: DataFrame with features and actual outcomes sensitive_attr: name of sensitive attribute column """ X = df_prod.drop(columns=['outcome', sensitive_attr] if sensitive_attr in df_prod.columns else ['outcome']) y_true = df_prod['outcome'] y_pred = model.predict(X) metrics = { 'timestamp': datetime.now().isoformat(), 'n_records': len(df_prod), 'overall_accuracy': (y_pred == y_true).mean(), 'group_metrics': {} } # Compute per-group accuracy if sensitive_attr in df_prod.columns: for group in df_prod[sensitive_attr].unique(): mask = df_prod[sensitive_attr] == group group_acc = (y_pred[mask] == y_true[mask]).mean() group_approval = y_pred[mask].mean() metrics['group_metrics'][str(group)] = { 'accuracy': group_acc, 'approval_rate': group_approval, 'count': int(mask.sum()) } return metrics # Load production data (simulated) df_prod = pd.read_csv('production_week_1.csv') # Compute metrics metrics = compute_fairness_metrics(df_prod, model, 'gender') # Check against thresholds THRESHOLDS = { 'min_accuracy': 0.82, 'max_approval_gap': 0.05, # 5% max gap between groups 'disparate_impact': 0.80 # 80% rule } # Alert if thresholds violated alerts = [] if metrics['overall_accuracy'] < THRESHOLDS['min_accuracy']: alerts.append(f"⚠ Accuracy dropped below {THRESHOLDS['min_accuracy']}") # Check approval rate gap if len(metrics['group_metrics']) >= 2: approval_rates = [m['approval_rate'] for m in metrics['group_metrics'].values()] gap = max(approval_rates) - min(approval_rates) if gap > THRESHOLDS['max_approval_gap']: alerts.append(f"⚠ Approval rate gap {gap:.1%} exceeds threshold {THRESHOLDS['max_approval_gap']:.1%}") if alerts: print("🚨 FAIRNESS ALERTS:") for alert in alerts: print(f" {alert}") else: print("✓ All fairness metrics within thresholds") # Log metrics with open('fairness_log.jsonl', 'a') as f: f.write(json.dumps(metrics) + '\n') print(f"\nMetrics logged to fairness_log.jsonl")
